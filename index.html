The Protected Companion-Robot RaaS Experience

Five scenes covering service-triggered exercise support, user-state-aware timing, outdoor mode switching, and robot-side guidance, shielding, and illumination behaviors.

SCENE 1

Context-Aware Trigger & Timing

UX Value

A health service can request that the robot prompt exercise — but not every moment is the right one to ask. The robot holds the request until the user is in a state to respond. Timing alone can turn the same prompt into a fundamentally different outcome.

Granted Claim Basis

A dual-server architecture. The Service Provider's server evaluates biometric data to issue an exercise request. The Robotics Operator's server temporarily holds this request, acquiring dynamic state data from the robot's sensors, and only transmits the execution instruction to the robot after determining the user's state is suitable.

SCENE 2

Physical Engagement

UX Value

A health service can send a digital exercise prompt — but most are ignored. When activity remains insufficient, the robot moves to the user and invites her to exercise in person through physical gestures. A request from the user's familiar companion robot is fundamentally different from a digital nudge that can be swiped away.

Granted Claim Basis

Triggered automatically when a prior digital message fails to achieve the target exercise amount. The robot utilizes optical sensors to detect the user's 3D position, navigates to them, executes physical gestures via actuators, and monitors user behavior via audio/visual sensors to accompany the workout.

SCENE 3

Automatic outdoor-mode transition when configured conditions are met

UX Value

A safety service may need to support the user 24/7, both inside and outside the home. Once the robot is outside with the user, it shifts into an outdoor mode that stays close, reads the surroundings, and helps keep the user safe. This creates adaptive safety support.

Granted Claim Basis

Protected via a comprehensive dual-architecture. The environment transition can be triggered via server commands, or autonomously by the robot itself. Upon crossing the threshold, the robot independently shifts from an indoor to an outdoor specification, e.g., dynamically increasing permitted mobility speeds.

SCENE 4

Proactive Guiding & Shielding

UX Value

A digital safety service can ease user anxiety, but cannot physically address danger in surroundings. The robot proactively steps in front to guide the user away from hazards, or physically interposes itself as a shield against threats. Physical intervention in the real world, rather than a digital alert, creates peace of mind.

Granted Claim Basis

Continuously monitors from a default standard relative position. Upon detecting a dangerous object in the movement direction or a moving body entering an alert area, it executes dynamic path recalculation. It then autonomously drives to a calculated "lead position" to guide routing, or a "guard position" directly between the user and the moving hazard.

SCENE 5

Illuminated Hazard Avoidance

UX Value

Conventional audio alerts or screen notifications require the user to process the information and decide how to react. Instead, the robot silently redirects its path-lighting to steer the user's steps away from obstacles, or projects warning lights on the ground to signal approaching danger. Guiding the user's movement visually on the ground induces a natural, almost subconscious evasion reflex.

Granted Claim Basis

The robot illuminates a nearby ground area. When sensors identify a dangerous object ahead, it dynamically controls the illumination device to irradiate a new ground area in a different direction, inducing a change in movement. For approaching moving objects, it projects light specifically onto a notification area on the road surface between the user and the threat.
